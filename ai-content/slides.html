Part I

The AI Revolution

From Turing to Transformers

The Long Road to AI

1950 Turing Test — Alan Turing asks "Can machines think?" and proposes the imitation game
1956 Dartmouth Conference — The term "Artificial Intelligence" is coined. Optimism is sky-high
1966 ELIZA — First chatbot. Pattern matching, no understanding — but fooled many people
1974-93 AI Winters — Overpromise, underdeliver. Funding dries up. Twice.
1997 Deep Blue beats Kasparov — Brute force search, not learning. But a landmark moment.

The Deep Learning Revolution

2012 AlexNet — Deep neural nets crush ImageNet. GPU training changes everything. The deep learning era begins.
2013 Word2Vec — Words become vectors. "King - Man + Woman = Queen." The foundation for embeddings.
2017 "Attention Is All You Need" — Google's Transformer paper. The architecture behind every modern LLM.
2018 GPT-1 & BERT — Pre-trained language models. Transfer learning comes to NLP.
2020 GPT-3 — 175B parameters. Few-shot learning. "It can write code?!"

The Generative AI Explosion

Nov 2022 ChatGPT launches — 100M users in 2 months. AI becomes mainstream overnight.
2023 GPT-4, Claude, Gemini, Llama 2 — Multimodal, open-source, tool use. An arms race begins.
2024 Agents & MCP — AI that uses tools, browses the web, writes code. Model Context Protocol standardizes integration.
2025 Spring AI 1.0 — The Spring ecosystem embraces AI. Java developers join the party.
2026 Spring AI 2.0 + Spring Boot 4 — Production-grade AI with observability, MCP, and agentic patterns. That's us, today.

Approaches to AI — What Changed?

Rule-Based (1960s-90s)

Hand-written rules, expert systems, decision trees. Brittle. Doesn't scale. Every edge case = new rule.

if city == "Berlin" then weather = ...

Classical ML (2000s-10s)

Learn patterns from data. SVM, random forests, gradient boosting. Good for structured data. Requires feature engineering.

features → model.fit() → predict

Generative AI (2020s)

Pre-trained on the internet. Understands language. Generates text, code, images. You describe what you want in natural language.

"Write a REST controller for..."

The paradigm shift: from programming rules to describing intent. The model figures out the rest.

How LLMs Actually Work

  • A Transformer neural network trained on vast text corpora
  • Core task: predict the next token given all previous tokens
  • Not a database — it generates text probabilistically
  • Stateless: every call is independent. Context must be provided each time.

Tokens

LLMs see tokens, not words. ~1 token ≈ 0.75 English words. "Spring Framework" → 2-3 tokens.

Context Window

Max tokens per request (input + output). GPT-4o: 128K. Claude: 200K. Qwen3: 32K.

It's Not Just Python Anymore

The Python Assumption

  • AI has been Python-first since the deep learning revolution
  • TensorFlow, PyTorch, scikit-learn, LangChain, HuggingFace — all Python
  • Java developers were told: "Learn Python or get left behind"
But here's the thing: building AI-powered applications is fundamentally different from training AI models.

Training vs. Consuming

GPU clusters, CUDA, tensor math, loss functions, gradient descent. This IS Python territory.

You probably won't do this.

vs

Building AI Applications

REST APIs, dependency injection, transaction management, security, observability, deployment. This is your territory.

This is what this workshop is about.

LLMs are accessed via HTTP APIs. You don't need Python to call an API. You need good application architecture — and that's where Java and Spring shine.

Meet Spring AI

The Spring Way to Build AI Applications

What Is Spring AI?

  • A Spring ecosystem project for AI-powered applications
  • Portable abstractions: write once, run against any provider
  • Auto-configuration: add a starter, set an API key, go
  • POJO-centric: your business logic stays in plain Java
Think of it as "Spring Data for AI" — a uniform API across providers, just like Spring Data across databases.

Stage 1

Chat Fundamentals

ChatModel, ChatClient, Prompts

The Simplest AI Call

GET /chat/01/joke

private final ChatModel chatModel;

@GetMapping("joke")
public String getJoke() {
    return this.chatModel.call("Tell me a joke");
}
ChatModel.call(String) — string in, string out. The absolute minimum API.

ChatClient — The Fluent API Recommended

GET /chat/02/client/joke?topic=java

// Full response chain
ChatResponse response = chatClient.prompt()
    .user("Tell me a joke about " + topic)
    .call()
    .chatResponse();
Generation generation = response.getResult();
AssistantMessage msg = generation.getOutput();
return msg.getText();

// Shorthand (preferred!)
String joke = chatClient.prompt()
    .user("Tell me a short joke")
    .call()
    .content();

ChatClient is to ChatModel what Spring Data JPA is to JDBC.

Prompt Templates

GET /chat/03/joke?topic=cats

// Inline template with {variable} placeholders
chatClient.prompt()
    .user(u -> u.text("Tell me a joke about {topic}")
                .param("topic", topic))
    .call().content();

// File-based template (classpath resource)
chatClient.prompt()
    .user(u -> u.text(new ClassPathResource("prompts/plays.st"))
                .param("author", author))
    .call().content();
Always use {variable} + .param() instead of string concatenation. Cleaner, testable, supports external files.

System Roles

GET /chat/06/fruit vs GET /chat/06/veg

public RoleController(ChatClient.Builder builder) {
    this.chatClient = builder
        .defaultSystem("""
            You are a helpful expert on plants, however
            you are not allowed to answer any question about
            vegetables - you can only answer about fruits.
            """)
        .build();
}

@GetMapping("/fruit")
public String fruit() {
    return chatClient.prompt()
        .user("What is the color of a banana?")
        .call().content();  // "A banana is yellow when ripe"
}

@GetMapping("/veg")
public String veg() {
    return chatClient.prompt()
        .user("What is the color of a carrot?")
        .call().content();  // "Sorry, I can only answer about fruits"
}

Structured Output & Tools

Type-Safe AI Responses & Function Calling

Structured Output

GET /chat/04/plays/object — AI → typed Java records

// Define your target type
record Play(String author, String title, Integer publicationYear) {}

// Return a List<String>
chatClient.prompt()
    .user(u -> u.text("List plays by {author}").param("author", "Shakespeare"))
    .call()
    .entity(new ListOutputConverter(new DefaultConversionService()));

// Return typed Java records — the Spring AI magic
Play[] plays = chatClient.prompt()
    .user(u -> u.text("List plays by {author}").param("author", "Shakespeare"))
    .call()
    .entity(Play[].class);
.entity(Class) auto-generates JSON schema, appends format instructions, and deserializes. No manual parsing.

Tool Calling with @Tool

GET /chat/05/time?city=Berlin

// Your Java methods become AI-callable tools
public class TimeTools {
    @Tool(description = "Returns the current time in a specific timezone")
    public String currentTimeIn(
            @ToolParam(description = "IANA timezone") String timeZone) {
        ZonedDateTime now = ZonedDateTime.now(ZoneId.of(timeZone));
        return "Today is " + now.getDayOfWeek() + " " + now;
    }
}

// Register tools with the prompt
chatClient.prompt()
    .tools(new TimeTools())
    .user(u -> u.text("What is the current time in {city}?")
                .param("city", city))
    .call().content();
The AI decides when to call the tool. Spring AI handles the multi-turn conversation automatically.

Stages 2 & 3

Embeddings & Vector Stores

The Foundation of Semantic Search

What Are Embeddings?

Dense float vectors that capture semantic meaning.

"king"   → [0.12, -0.45, 0.78, 0.33, ...]  ┐
                                                ├ close (cosine ~ 0.85)
"queen"  → [0.14, -0.42, 0.75, 0.31, ...]  ┘

"banana" → [-0.67, 0.22, 0.11, -0.89, ...]    far away (~ 0.15)
// Generate an embedding
float[] vector = embeddingModel.embed("mountain bike");
// → 768 floats (nomic-embed-text) or 1536 (OpenAI)

Chunking, Reading & Storing

// Large documents must be split before embedding
TokenTextSplitter splitter = new TokenTextSplitter();
List<Document> chunks = splitter.split(largeDocument);

// Read from various sources
JsonReader jsonReader = new JsonReader(bikesResource,
    "name", "price", "shortDescription", "description");

// Load into vector store
vectorStore.add(splitter.split(jsonReader.get()));

// Semantic similarity search
List<Document> results = vectorStore
    .similaritySearch("mountain bike");

SimpleVectorStore

In-memory, zero setup. Great for dev/testing.

PgVectorStore

PostgreSQL + pgvector. HNSW index. Production-ready.

Stage 4

RAG & Chat Memory

Grounding AI in Your Data

RAG — Retrieval Augmented Generation

The most important AI pattern for enterprise applications.

Embed docs → Search → Augment prompt → Generate

Manual RAG

List<Document> docs = vectorStore
    .similaritySearch(query);
String context = docs.stream()
    .map(Document::getText)
    .collect(joining("\n"));

chatClient.prompt()
  .system(s -> s.text(qaPrompt)
    .param("context", context))
  .user(query)
  .call().content();

QuestionAnswerAdvisor

// All-in-one: search + augment
chatClient.prompt()
  .advisors(
    new QuestionAnswerAdvisor(
        vectorStore))
  .user(query)
  .call()
  .content();

// Advisor automatically:
// 1. Searches vector store
// 2. Injects into prompt
// 3. Generates answer

Chat Memory

GET /mem/02/hello?message=Hi, I'm Alice → GET /mem/02/name

ChatMemory memory = MessageWindowChatMemory.builder()
    .chatMemoryRepository(new InMemoryChatMemoryRepository())
    .maxMessages(10)
    .build();

this.chatClient = builder
    .defaultAdvisors(PromptChatMemoryAdvisor.builder(memory).build())
    .build();

// Request 1: "Hi, I'm Alice" → stored in memory
// Request 2: "What's my name?" → AI remembers "Alice"
chatClient.prompt()
    .user(message)
    .advisors(a -> a.param("chat_memory_conversation_id", id))
    .call().content();
LLMs are stateless. The memory advisor re-injects conversation history with every request.

Advanced Patterns & MCP

Chain of Thought, Self-Reflection, Model Context Protocol

Chain of Thought

GET /cot/bio/flow — multi-step pipeline

public List<String> writeBio(String profile) {
    // Step 1: Create outline
    String outline = chatClient.prompt()
        .user(u -> u.text("Create an outline for a biography:\n\n{profile}")
                    .param("profile", profile))
        .call().content();

    // Step 2-3: Draft and condense (similar pattern)

    // Step 4: Refine with expert persona
    String refined = chatClient.prompt()
        .system("You are an expert biography editor")
        .user(u -> u.text("Polish this biography:\n\n{draft}")
                    .param("draft", draft))
        .call().content();

    return List.of(outline, draft, condensed, refined);
}

Each step focuses on one aspect. The .system() prompt changes the AI's role per step.

Self-Reflection Agent

GET /reflection/bio/agent?iterations=3

Writer → Critic → Writer (revise) → Critic → Final 
for (int i = 0; i < iterations; i++) {
    // Writer generates (or revises based on feedback)
    String draft = chatClient.prompt()
        .system("You are a professional biography writer")
        .user(writerPrompt + feedback)
        .call().content();

    // Critic reviews
    String feedback = chatClient.prompt()
        .system("You are a harsh but fair biography critic")
        .user("Review this biography:\n\n" + draft)
        .call().content();
}

Stage 7

Agentic Systems

AI That Controls Its Own Execution

From Calls to Agents

Simple Call

You control the flow. One prompt, one response. Stages 1-4.

Developer decides what happens.

Pipeline

You orchestrate multi-step chains. CoT, reflection. Stage 5.

Developer designs the steps.

Agent

The model controls execution. Decides tools, steps, when to stop. Stage 7.

Model decides what happens.

Inner Monologue Agent

The model reveals its reasoning via a structured tool call.

// Force the model to always respond via the tool
OpenAiChatOptions options = OpenAiChatOptions.builder()
    .toolChoice("required")
    .build();

// The tool captures visible message AND private reasoning
@Tool(description = "Send a message to the user")
public ChatResponse sendMessage(
    @ToolParam(description = "The message") String message,
    @ToolParam(description = "Private thoughts") String innerThoughts) {
    return new ChatResponse(message, innerThoughts);
}

Model-Directed Loop

The model decides whether to continue processing or stop.

@Tool(description = "Send a message to the user")
public ChatResponse sendMessage(
    @ToolParam(description = "Message") String message,
    @ToolParam(description = "Reasoning") String innerThoughts,
    @ToolParam(description = "Continue thinking?") boolean requestReinvocation) {
    return new ChatResponse(message, innerThoughts, requestReinvocation);
}

// The agent loop
List<ChatResponse> trace = new ArrayList<>();
for (int step = 0; step < MAX_STEPS; step++) {
    ChatResponse response = chatClient.prompt()
        .options(OpenAiChatOptions.builder().toolChoice("required").build())
        .tools(new SendMessageTool())
        .user(step == 0 ? userMessage : "Continue.")
        .call().content();

    trace.add(response);
    if (!response.requestReinvocation()) break;  // Model says "done"
}
